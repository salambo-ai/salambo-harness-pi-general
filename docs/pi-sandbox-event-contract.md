# PI Sandbox Event Contract

This document is the source of truth for the raw event stream emitted by this sandbox template.

Important boundary:

- this sandbox emits internal transport events for the platform
- it does not emit the final client-facing SSE contract directly
- when these events pass through the Salambo platform, the platform adapts them into `/v1/responses` streaming events

So:

- sandbox repo owns `sandbox.run.*` and `session.event`
- platform/app owns `response.*`, `salambo:agent.*`, `salambo:raw.pi.event`, and `[DONE]`

For the full PI inner event catalog, see [pi-agent-event-catalog.md](/C:/Users/nicol/WebstormProjects/salambo-sandbox/salambo-sandbox-pi-sdk/docs/pi-agent-event-catalog.md).

The platform-side SSE mapping belongs in the platform/app repo, not this template repo.

## Ownership

The sandbox owns:

- the HTTP control surface in `src/routes/agent.ts`
- the outer lifecycle events emitted by `src/core/agent-runner.ts`
- the raw `session.event` envelope emitted by `src/core/event-store.ts`

The sandbox does not own:

- the public `/v1/responses` SSE shape
- Open Responses compatibility events
- client-facing projection semantics

## Event Families

The sandbox emits exactly these payload families:

- `sandbox.run.init`
- `sandbox.run.ready`
- `sandbox.run.complete`
- `sandbox.run.cancelled`
- `sandbox.run.error`
- `session.event`

The sandbox does not emit `response.*` or `salambo:*` SSE events itself.

## Transport Backends

The same payloads are produced regardless of backend:

- in S2 mode, each payload is appended as one JSON record with `event-type = payload.type`
- in local mode, the payload is stored in memory and exposed via `GET /agent/events/:sandboxId`

Local fallback wraps each payload like this:

```json
{
  "sequence": 7,
  "streamName": "salambo:sandbox-123",
  "payload": {
    "type": "session.event",
    "sandboxId": "sandbox-123",
    "sessionId": "pi-session-123",
    "event": {},
    "timestamp": "2026-04-10T10:00:00.000Z"
  }
}
```

The `sequence` number is transport metadata from the local event store. It is not embedded in the payload itself.

## Common Rules

- `sandboxId` is the outer platform/sandbox run identifier
- `sessionId` is the inner PI session identifier when known
- `timestamp` is always an ISO 8601 string generated by the sandbox at emit time
- payloads are JSON-sanitized before emission
- `bigint` values are stringified during sanitization

## Event Payloads

### `sandbox.run.init`

Emitted first for every run attempt.

```json
{
  "type": "sandbox.run.init",
  "sandboxId": "sandbox-123",
  "workspace": "/workspace",
  "promptPreview": "User prompt preview",
  "metadata": {},
  "timestamp": "2026-04-10T10:00:00.000Z"
}
```

Notes:

- `promptPreview` is truncated to 2000 characters
- `metadata` is whatever the caller supplied to `/agent/query`, after JSON sanitization

### `sandbox.run.ready`

Emitted once the sandbox has a stable PI session id for the run.

```json
{
  "type": "sandbox.run.ready",
  "sandboxId": "sandbox-123",
  "sessionId": "pi-session-123",
  "timestamp": "2026-04-10T10:00:01.000Z"
}
```

Notes:

- for resumed runs, `sessionId` is the stable resumed session id
- `sandbox.run.ready` is emitted before any `session.event`

### `sandbox.run.complete`

Emitted when the prompt finishes without cancellation.

```json
{
  "type": "sandbox.run.complete",
  "sandboxId": "sandbox-123",
  "sessionId": "pi-session-123",
  "timestamp": "2026-04-10T10:00:05.000Z"
}
```

### `sandbox.run.cancelled`

Emitted when the run abort signal wins.

```json
{
  "type": "sandbox.run.cancelled",
  "sandboxId": "sandbox-123",
  "sessionId": "pi-session-123",
  "timestamp": "2026-04-10T10:00:05.000Z"
}
```

Notes:

- `sessionId` may be absent if cancellation happens before session creation completes

### `sandbox.run.error`

Emitted when the run fails for a non-cancellation reason.

```json
{
  "type": "sandbox.run.error",
  "sandboxId": "sandbox-123",
  "sessionId": "pi-session-123",
  "error": {
    "message": "Something failed",
    "name": "Error",
    "stack": "..."
  },
  "timestamp": "2026-04-10T10:00:05.000Z"
}
```

Notes:

- `sessionId` may be absent if the failure happens before session creation completes
- `error` is omitted on `sandbox.run.cancelled`

### `session.event`

Emitted for every PI runtime event received from `session.subscribe(...)`.

```json
{
  "type": "session.event",
  "sandboxId": "sandbox-123",
  "sessionId": "pi-session-123",
  "event": {
    "type": "message_update",
    "assistantMessageEvent": {
      "type": "text_delta",
      "contentIndex": 0,
      "delta": "hello"
    }
  },
  "timestamp": "2026-04-10T10:00:02.000Z"
}
```

Notes:

- `event` is the sanitized raw PI event object
- the sandbox intentionally does not reinterpret it into product-specific message types
- ordering matches the order received from the PI subscription callback

## What Can Appear Inside `session.event.event`

The sandbox forwards PI `AgentSession.subscribe()` events as-is. Common top-level values include:

- `agent_start`
- `agent_end`
- `turn_start`
- `turn_end`
- `message_start`
- `message_update`
- `message_end`
- `tool_execution_start`
- `tool_execution_update`
- `tool_execution_end`
- `queue_update`
- `compaction_start`
- `compaction_end`
- `auto_retry_start`
- `auto_retry_end`

`message_update` may itself contain nested `assistantMessageEvent` deltas such as:

- `text_start`
- `text_delta`
- `text_end`
- `thinking_start`
- `thinking_delta`
- `thinking_end`
- `toolcall_start`
- `toolcall_delta`
- `toolcall_end`
- `done`
- `error`

See [pi-agent-event-catalog.md](/C:/Users/nicol/WebstormProjects/salambo-sandbox/salambo-sandbox-pi-sdk/docs/pi-agent-event-catalog.md) for the complete installed-package catalog.

## Ordering Guarantees

Within one sandbox run, the emitted payload order is:

1. `sandbox.run.init`
2. `sandbox.run.ready`
3. zero or more `session.event`
4. exactly one terminal event:
   - `sandbox.run.complete`, or
   - `sandbox.run.cancelled`, or
   - `sandbox.run.error`

Additional notes:

- `sandbox.run.ready` is emitted before the prompt is dispatched
- `session.event` payloads are serialized through one promise chain before append, so the sandbox preserves PI callback order
- after the terminal event, the sandbox runner does not intentionally emit more payloads for that run

## Non-Goals

This sandbox contract intentionally does not define:

- OpenAI Responses event names
- final SSE framing
- `[DONE]`
- client-side tool item reconstruction
- reasoning-summary semantics

Those belong to the platform adapter layer, not the sandbox.
